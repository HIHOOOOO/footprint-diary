### adb-toss 전역 별칭 등록

에뮬레이터 연결 시마다 `adb reverse tcp:8081 tcp:8081 && adb reverse tcp:5173 tcp:5173`를 직접 치는 게 번거로워서 별칭으로 등록했다.

처음엔 `package.json`의 `scripts`에 `npm run adb`로 추가했는데, 사용자가 "이 프로젝트에서만 쓰는 게 아니라 AIT 앱 전반에서 써야 하지 않냐"고 지적. 맞는 말 — AIT 앱은 모두 동일한 포트를 쓰니까 전역 별칭이 적합하다.

`package.json`에서 제거하고 `~/.bashrc`에 `alias adb-toss='...'`로 등록. 터미널 재시작 또는 `source ~/.bashrc` 후 사용 가능.

### SDK 2.0.1 마이그레이션

토스 AIT 팀에서 2026년 3월 23일까지 SDK 2.0.1 마이그레이션을 강제한다는 공지를 확인했다. 3월 30일 토스앱이 React Native 0.84 기반으로 업데이트되는데, 1.x 번들은 RN 0.84와 호환이 보장되지 않아서 기한 이후엔 콘솔 업로드 자체가 막힌다.

이 프로젝트는 WebView 앱이라 마이그레이션 절차가 단순하다. WebView 경로는 두 가지만 하면 된다:

1. `@apps-in-toss/web-framework`를 2.0.1로 올리기
2. `package.json` build 스크립트를 `granite build` → `ait build`로 교체

`npm install --legacy-peer-deps` 후 `npm run build` 실행. `ait build`가 기존 RN 0.72.6 번들에 더해 **RN 0.84.0 번들을 추가로 생성**하는 것을 확인했다. `footprint-diary.ait` 아티팩트 생성까지 정상.

### 제출 전 QA 및 버그 수정

번들 제출 전 코드 전체를 검토했다. 발견된 문제 세 가지를 수정했다.

**달력 요일 표기 오류** — 미니 달력의 요일 헤더가 포르투갈어(`D, S, T, Q, Q, S, S`)로 되어 있었다. 언제부터 이랬는지는 불명확하나, 한국 앱이니 한국어로 수정하는 것이 당연. 처음엔 `일월화수목금토`(일요일 시작)로 바꾸려 했는데, 사용자가 "토스앱 달력은 월요일 시작이지 않냐"고 확인 질문. 맞는 지적이라 `월화수목금토일`(월요일 시작)로 변경. blanks 계산도 `(getDay() + 6) % 7`로 함께 수정 — 일요일(0)이 6번째 칸(토요일 다음)에 오도록.

**GPS 동의 다이얼로그 버튼 순서 반전** — TDS `ConfirmDialog`의 `cancelButton`에 "허용할게요"가, `confirmButton`에 "허용 안 할게요"가 배치되어 있었다. prop 이름의 의미와 실제 동작이 뒤바뀐 상태. `cancelButton`에 "허용 안 할게요", `confirmButton`에 "허용할게요"로 스왑.

**버전 번호** — `package.json`의 `version`이 `0.0.0` 상태였다. 제출본이니 `1.0.0`으로 올렸다.

수정 후 `ait build` 빌드 완료.

### 내비게이션 바 브랜드 로고 추가

앱 상단에 브랜드 로고가 보이지 않는다는 피드백이 들어왔다. 현재 헤더 구조를 보면 날짜 pill이 화면 중앙에 떠 있고, 햄버거 메뉴 버튼이 좌상단에 따로 존재하는 구조인데, 로고가 어디에도 없었다.

`granite.config.ts`의 `brand.icon`으로 등록된 `/footprint-icon-mono.svg`를 재사용하기로 했다. 별도 파일 추가 없이 이미 있는 아이콘을 활용하는 게 자연스럽다.

햄버거 버튼 바로 오른쪽에 동일한 원형 pill 스타일(`bg-white/90 backdrop-blur shadow-sm rounded-full border border-gray-100`)의 컨테이너를 추가하고 그 안에 `<img src="/footprint-icon-mono.svg" />` 삽입. 햄버거 버튼과 시각적으로 세트처럼 보이도록 `flex items-center gap-2`로 묶었다.

이후 `ait build` → `footprint-diary.ait` 생성 완료.

## 2026-03-13

### 브랜드 로고 반려 원인 파악 및 수정

토스 검수에서 "공통 내비게이션 바에 브랜드 로고가 표시되지 않아요"로 반려됐다. 이전 세션에서 사이드바 헤더에 `<img>` 태그를 잘못 추가했고, 그게 깨진 이미지로 표시되고 있었다. 사이드바 헤더가 아니라 Toss 시스템 네이티브 nav bar에 로고가 필요한 것이었다.

`granite.config.ts`의 `brand.icon`이 네이티브 nav bar 로고를 제어한다. 기존에 SVG로 설정돼 있었는데, 네이티브 레이어에서 SVG를 제대로 처리 못할 수 있으므로 PNG로 변환했다. `sharp-cli`로 `footprint-icon-mono.svg` → `footprint-icon-mono.png` (96×96) 변환 후 `brand.icon`을 PNG 경로로 교체. `Footprint.jsx`의 잘못된 `<img>` 태그는 이미 없는 상태였다. `ait build` 재실행으로 번들 갱신.

---

### 샌드박스 앱 / 에뮬레이터 전쟁

브랜드 로고가 실제로 nav bar에 뜨는지 확인하려고 샌드박스 테스트를 시도했는데, 여기서 지옥이 시작됐다.

**에뮬레이터 저장 공간 부족** — 최신 샌드박스 APK(`rn-miniapp-real-release-protected-1.apk`) 설치 시 `INSTALL_FAILED_INSUFFICIENT_STORAGE`. AVD 설정에서 Internal storage를 10GB로 올리고 Wipe Data 했는데, Medium_Phone AVD 자체가 Android Studio 목록에서 사라졌다. `avdmanager` 명령으로 새 AVD(`Pixel_API36_10GB`)를 만들었지만 에뮬레이터가 블랙 스크린에서 부팅 실패를 반복했다. 원인을 추적하니 `Medium_Phone.ini` 파일이 없어진 것. 복원하고 Wipe Data + Cold Boot로 Medium_Phone 재기동 성공.

**ARM64 vs x86_64 호환성** — Medium_Phone에 샌드박스 APK 설치 후 실행 시 `libtcj.so is for EM_AARCH64 instead of EM_X86_64` 에러로 즉시 크래시. 새 샌드박스 APK가 ARM64 전용으로 빌드됐고, x86_64 에뮬레이터에서는 Toss 보안 라이브러리가 번역 레이어를 거부해서 실행 불가. 에뮬레이터로는 테스트 자체가 불가능한 구조적 문제.

**실물 기기 연결** — USB로 Samsung Galaxy 연결 후 APK 설치 성공. 그런데 이번엔 `StackOverflowError: stack size 8188KB`로 크래시. SDK 2.0.x 마이그레이션 FAQ에 나오는 "메트로 서버 연결 문제" 증상과 일치. `adb reverse` 재적용 후 재시도했으나 샌드박스 앱 자체 **로그인 실패**로 진행 중단.

**adb-toss 별칭 업데이트** — 기존 별칭이 시리얼 번호 없이 작동해서 에뮬레이터+실기기 동시 연결 시 어느 쪽에 적용될지 불확실했다. `while read dev`로 모든 연결 기기에 순차 적용하도록 `~/.bashrc` 업데이트.

**다음 세션 프롬프트:**
> `샌드박스 앱 실기기 로그인 실패 이어서 해결. 실기기(RFCX60HV9SZ) 연결 상태에서 npm run dev → adb-toss → 샌드박스 앱 로그인 시도. 로그인 안 되면 토스 채널톡 문의. SDK 2.0.1 → 2.0.5 업그레이드도 함께 진행.`

## 2026-03-14

### 실기기 샌드박스 로그인 성공

이전 세션 마지막 문제였던 샌드박스 로그인 실패를 이어서 해결했다. 결론부터 말하면 로그인 자체 문제가 아니라 adb reverse가 실기기가 아닌 에뮬레이터에 연결됐던 것이었다.

`adb -s RFCX60HV9SZ reverse` 명령이 "device not found"로 실패했고, 그 상태에서 `adb reverse`(시리얼 없는 명령)를 쓰니 기본 연결 대상인 x86_64 에뮬레이터에 adb reverse가 적용됐다. 샌드박스 앱은 실기기에서 실행 중이었으므로 localhost:8081에 도달할 수 없었던 것.

실기기를 USB로 다시 꽂고 신뢰 허용 후 `adb -s RFCX60HV9SZ reverse tcp:8081 tcp:8081 && adb -s RFCX60HV9SZ reverse tcp:5173 tcp:5173` 실행. 샌드박스 앱에서 `http://localhost:8081` 입력하니 앱이 정상 로드됐다. FCP 약 750ms, LCP 약 600ms로 빠른 편.

### 에뮬레이터 크래시 원인 파악

logcat에서 `libtcj.so is for EM_AARCH64 instead of EM_X86_64` 에러를 확인. 샌드박스 앱 자체가 ARM64 전용 빌드라서 x86_64 에뮬레이터에서는 구조적으로 실행 불가. 에뮬레이터로 테스트하려면 arm64-v8a AVD를 별도로 생성해야 한다.

### 공통 네비게이션 바 브랜드 로고 미표시 해결

실기기에서 앱은 정상 동작했지만, 토스 공통 네비게이션 바에 브랜드 로고가 뜨지 않았다. 이 문제는 이전 검수에서 반려 사유이기도 했다.

원인 추적 과정이 길었다.

`granite.config.ts brand.icon`에 `data:image/png;base64,...` 형식의 값이 들어 있었다. `.ait` 번들 헤더를 직접 파싱해보니 `brandIcon` 필드에 그 base64 문자열이 그대로 저장돼 있었다. 네이티브 토스 앱이 nav bar 아이콘을 그릴 때 data URI를 처리하지 못하는 것이 문제였다.

해결 방향은 **CDN URL**로 교체하는 것이었다. AIT 콘솔에서 브랜드 아이콘을 등록하면 `https://static.toss.im/appsintoss/...` 형태의 영구 URL이 발급된다. 사용자가 해당 URL을 확인해서 공유해줬고, `brand.icon`에 그 URL을 넣었다.

그런데 교체 직후에도 "여전히 안 뜬다"는 피드백이 왔다. 이유는 **`npm run dev`를 재시작하지 않아서**였다. granite dev 서버는 시작 시점에 `granite.config.ts`를 읽어 runtime setup script를 생성한다. 서버가 살아있는 상태에서 config를 바꿔도 구 설정이 계속 주입되고 있었던 것. dev 서버 재시작 후 샌드박스 앱을 다시 열자 nav bar에 브랜드 로고가 정상 표시됐다.

정리하면 두 가지 문제가 겹쳐 있었다: (1) data URI는 네이티브 nav bar가 렌더링 못 함, (2) config 변경 후 dev 서버 재시작 필수.

CDN URL 방식이 Toss 공식 권장 방법인지는 문서에 명시가 없어서 불확실하다. 채널톡 문의를 통해 공식 확인이 필요하다.
